# 阶段五第一步：翻译浮窗接入 Provider 输出

阶段五第一步用于验证 Zotero AnnotAI 可以把 PDF 选区文本发送给已配置的 OpenAI-compatible Provider，并在 `翻译` 浮窗中显示真实结果。这个阶段仍不创建高亮、不写入批注，`解释` 和 `问答` 继续保持诊断壳。

0.1.19 已进入阶段五第二步：翻译完成后可手动点击 `写入批注`，但翻译请求返回时仍不会自动写批注或自动高亮。第二步说明见 `docs/phase-5-translation-annotation-writeback.md`。

## 测试包

```text
dist/zotero-annotai-0.1.19.xpi
```

## 翻译请求行为

- 默认目标语言固定为中文。
- 请求只发送选中文本和论文标题、作者、日期等元数据。
- 不发送前后文，不发送整篇 PDF 文本。
- 不记录 API key、Authorization header、完整 prompt 或完整选区文本。
- 每个请求使用当前 Provider 的 `timeoutMs`。

## 重复点击规则

- 同一 reader 内只保留一个翻译浮窗。
- 重复点击不改变已有浮窗的位置和大小。
- 旧翻译请求未返回时，如果用户对同一选区重复点击 `翻译`，插件不发送第二个请求。
- 旧翻译请求未返回时，如果用户选中新文本并点击 `翻译`，插件会尝试 abort 旧请求，并立即发送新文本请求。
- 如果旧请求无法真正取消，旧结果返回后也会按 requestID 丢弃，不覆盖新文本结果。

## 手动验收

1. 安装 `dist/zotero-annotai-0.1.19.xpi` 并重启 Zotero。
2. Debug Output 中确认出现 `Startup 0.1.19`。
3. 在设置页保存可用的 OpenAI-compatible Provider，并确认连接测试成功。
4. 打开 PDF，选中文本，点击 `翻译`。
5. 翻译浮窗应显示 `正在翻译...`，随后显示真实翻译结果、模型和耗时。
6. 请求未返回时重复点击同一选区的 `翻译`，不应发送第二个请求。
7. 请求未返回时切换新文本并点击 `翻译`，浮窗应立即切换到新文本，旧结果不应覆盖新结果。
8. 空 base URL 或空 model 应显示配置不完整。
9. 错误 API key 应显示认证失败，日志不泄露 key。
10. `解释` 和 `问答` 仍显示诊断内容，不调用 AI。

## 阶段三回归

- Zotero 原生按钮正常显示。
- Translate for Zotero 等其他插件入口正常显示。
- AnnotAI 三个按钮仍在 Zotero 原生文本选区弹窗内部。
- 三个 AnnotAI 浮窗可打开、拖动、缩放、关闭。
- 多个 AnnotAI 浮窗重叠时，点击内容区可以置顶。
- PDF 鼠标滚轮正常滚动。
